很多人问 Steve Hanov：“你为什么不融资？”

他的回答很直接：**不需要**。

![95f16b3dbf861a9156cb2c6d1be39f5e2fcfc1b1f4c5f11ee6f9e182bd4c5f5a.png](https://i.829259.xyz/api/rfile/95f16b3dbf861a9156cb2c6d1be39f5e2fcfc1b1f4c5f11ee6f9e182bd4c5f5a.png)

他一个人同时运营着多家**月收入稳定在1万美元以上**的小公司，而支撑这一切的底层技术栈，每个月只花 **20 美元**。

你可能会说：“啊？20 美元？连一台云数据库都不够吧？”

区别可太大了。

传统做法是：上 AWS、用 Kubernetes、搭微服务、买托管数据库……还没赚钱，账单先来了。Steve 以前也这么干过，后来发现——**绝大多数产品根本到不了需要那种架构的量级**。

现在他的做法是：**VPS + Go + SQLite + 本地AI**，全部塞进一台 5–10 美元/月的服务器里。

具体怎么搭？

- **服务器**：Linode 或 DigitalOcean 最低配，5–10 美元/月。不用 AWS/GCP，避免复杂计费和配置陷阱。
- **编程语言**：Go。编译成一个二进制文件，扔到服务器上就能跑，内存占用极低。
- **数据库**：SQLite，开启 **WAL 模式**。别小看它，中等并发完全扛得住，省掉一个独立的数据库服务器。
- **AI 能力**：本地跑模型（二手显卡 + Ollama/VLLM），避免按次付费的 API 高昂开销。再用 **OpenRouter** 兜底调用 GPT/Claude，自动降级切换。
- **开发辅助**：GitHub Copilot，以极低成本获得代码生成能力。

更贴心的是：他并不是盲目“省”。每个选择都有明确的理由——**减少运维负担、降低固定成本、一个人能维护多个产品**。

当然，这套方案有它的边界：不适合超大规模、超高并发的场景。但对于 SaaS、工具类产品、内容站点来说，**完全够用**。

还有他们画了个饼：**多代理协作**和**长期记忆**？哦不好意思，那是 Claude 托管代理的饼。但 Steve 这边其实也有延伸可能——未来本地 AI 模型越来越强，这套极简架构能做的事情只会更多。

一句话总结：如果你还在纠结上什么云、搭什么数据库、要不要搞微服务，可以歇一歇了。**先跑通业务、先盈利，再谈扩展**。用 20 美元的技术栈养活月入 1 万美元的产品，不是神话，是选择。

想详细了解的可以去搜 Steve Hanov 的原文。我已经在把自己的 side project 往这个方向迁移了，看看每月能省下多少成本。
